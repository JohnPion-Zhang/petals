Key Insight: The loop continues until either:
  • The LLM provides a final answer (no tool calls)
  • Max iterations are reached (default: 10)
  • An unrecoverable error occurs

Tool Parsing Syntax

LLM Output Format

LLMs output tool calls using a simple XML-like syntax:

# LLM Output: I'll check the weather for you.<tool_call>get_weather({"city": "Tokyo"})</tool_call>

ToolParser Extraction

The parser uses regex to extract tool calls:

# Pattern: <tool_call>(.*?)</tool_call> # Format: tool_name({...json args...})

Example: Multiple Tool Calls

# LLM Output: Let me help with that.<tool_call>get_weather({"city": "Tokyo"})</tool_call><tool_call>calculate({"expression": "15+27"})</tool_call> # Parsed to: [ ToolCall(name="get_weather", arguments={"city": "Tokyo"}), ToolCall(name="calculate", arguments={"expression": "15+27"}) ]

Key Design Decisions

1. litellm for Provider Abstraction

Unified interface for OpenAI, Anthropic, and custom endpoints:

from petals.client.http_client import HTTPClient client = HTTPClient( api_key="key", base_url="http://127.0.0.1:3456/v1", default_model="claude-4.5-sonnet", )
Why litellm?
  • Handles auth headers, timeouts, retries automatically
  • Provider prefix system (openai/, anthropic/)
  • Single interface for multiple LLM providers

2. Generator Pattern for Memory Efficiency

async for result in self.executor.parallel_execute(tool_calls): results.append(result)

Uses asyncio.as_completed() to yield results as they finish, reducing memory for large tool sets.

3. Anthropic /messages API Support

Fixed URL normalization for Anthropic-compatible endpoints:

# litellm adds /v1/messages, so strip it from base_url api_base = self.base_url.rstrip("/") if api_base.endswith("/v1"): api_base = api_base[:-3] # Remove "/v1"
